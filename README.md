Deterministic, version-controlled authentication threat hunting over a synthetic
enterprise dataset: seven SQL detections with planted ground truth, measured
precision/recall, ATT&CK tagging, static threat-intel enrichment, and a pytest +
GitHub Actions pipeline that fails when detection logic regresses.

is the manual precursor to (see [Severity model](#severity-model-and-the-path-to-adte)).

## Why this exists

A detection you cannot measure is a claim, not a control. Because every anomaly
here is planted, ground-truth labels exist (`data/ground_truth.sql`), so the
appendix reports **measured** precision and recall instead of adjectives. The
detections are versioned, tested, and CI-gated: when a rule's logic breaks, a
regression test goes red. That is the difference between a SQL worksheet and
detection-as-code.

## Phase 1: Baseline Enumeration

Before hunting anomalies, establish what "normal" looks like. All dates and
thresholds come from a `hunt_config` table (SQLite views can't take parameters),
so there are no magic literals scattered through the queries.

### 1.1 Geographic Login Baseline

One row per (user, country) from the baseline window — replacing the previous
`GROUP_CONCAT` + `INSTR` substring test, which was a fragile CSV-in-a-column hack.

```sql
SELECT DISTINCT username, country
FROM log_in_attempts
WHERE success = 1
  AND login_date < (SELECT value FROM hunt_config WHERE key = 'baseline_end');
```

Most users authenticate only from the US; E002 shows a documented US/CA travel
pattern that later serves as a tuning test for impossible travel. Full baseline
in [results/DETECTION_OUTPUT.md](results/DETECTION_OUTPUT.md).

### 1.2 Endpoint Patch Posture

Patch age is measured against `hunt_config.asof_date`, not a hardcoded date.

```sql
SELECT device_id, owner_name, days_since_patch, patch_risk
FROM v_patch_status WHERE patch_risk = 'CRITICAL';
```

Three endpoints exceed the 180-day threshold — SRV-001 (285d, a backup server),
DEV-011 (245d), DEV-012 (219d). DEV-011 also carries an ownership conflict
(Phase 4).

## Phase 2: Anomaly Detection

Seven detections. Each notes the concrete defect fixed relative to the previous
version. Every rule carries a MITRE ATT&CK technique; the deployed SQL is in
[sql/detection_queries.sql](sql/detection_queries.sql) and its live output in
[results/DETECTION_OUTPUT.md](results/DETECTION_OUTPUT.md).

### Rule 1 — Unfamiliar Country · `T1078`

Successful login from a country not in the user's baseline. Two explicit
branches: users **with** a baseline (not-in-allowlist test), and users **without**
one (new/dormant accounts — flag only the first success).

> **Fixed:** the old `known_countries IS NULL` test conflated a brand-new
> employee with a terminated account. The no-baseline branch now names that case
> explicitly ("new or dormant account"). Rule 1 also incidentally flags the
> terminated account E016 — but **Rule 6 owns that case**; Rule 1's catch is
> incidental and doesn't count toward its recall.

Flags E001 (CN), E007 (DE), E016 (RU, incidental), and — by design — the dormant
account E017 as a **benign false positive**. That single deliberate FP is why
Rule 1's measured precision is **0.75, not a suspicious 1.00**.

### Rule 2 — Impossible Travel · `T1078`

`LAG()` over each user's successful logins, velocity-gated against precomputed
country-centroid distances. A transition flags only when the **required velocity
exceeds 900 km/h** (commercial cruise) — "different country within N hours" alone
is not evidence.

> **Fixed:** the old rule used `country != country` as its only spatial test (so
> E002's legitimate US→CA travel would flag) and an `O(n²)` self-join ordered by
> `event_id`, not time. The rewrite computes required velocity, exposes it in
> every row, and is a single ordered pass — harmless at 161 rows, but the self-join
> was quietly catastrophic at SIEM scale.

| user | leg | hours | required km/h | verdict |
|---|---|---|---|---|
| E007 | DE → US | 4.3 | ~1,835 | flagged (impossible) |
| E001 | CN → US (return) | 10.8 | ~1,040 | flagged (over gate) |
| E001 | US → CN (outbound) | 13.3 | ~840 | **not** flagged (under gate) |
| E002 | US → CA | 5.5 | ~345 | **not** flagged (legitimate travel) |

**E001, calibrated:** the outbound US→CN leg (~840 km/h) is *under* the gate —
borderline-possible by velocity alone. It is the **return** leg (~1,040 km/h),
landing right after a 03:27 login, that tips the case. The numbers don't turn
this into a binary; they show exactly where the judgment lives. The US↔CA
corridor is additionally recorded in a `travel_exceptions` table as a documented
downgrade-to-MEDIUM (never a silent suppression), because centroid distance
overstates border-region travel.

### Rule 3 — After-Hours Login · `T1078`

Login outside the user's **own learned hour window** (baseline min/max ± 2h),
falling back to a department window, then a global default — not a hardcoded 9–5.
`baseline_source` shows which tier fired.

> **Fixed:** "02:30 is unusual for Legal" was previously asserted with no baseline
> behind it. The hour window is now learned from data, extending the same
> baselining idea already used for geography. After-hours stays a **weak signal**
> (never above MEDIUM); it earns its keep by corroborating stronger rules —
> e.g. it independently catches the attacker's 10:10 brute-force success on E004,
> which falls outside E004's learned window.

### Rule 4 — Brute Force (failures then success) · `T1110.001`

Failures within an hour before a success, **with a `NOT EXISTS` check that no
success intervenes**.

> **Fixed:** the old query counted failures in a window before a success but never
> verified the failures were uninterrupted, so one failure could be counted
> against multiple later successes. It also silently lacked the detection-window
> filter that rules 1–3 had. Both corrected.

Flags E004: 8 uninterrupted failures → success from **185.220.101.45**, which the
enrichment step names as a **Tor exit node** (see [Enrichment](#threat-intel-enrichment)).

### Rules 5a / 5b — Password Spray vs Failed Brute Force · `T1110.003` / `T1110.001`

The previous "password spray" rule grouped failures **by account**, which can
never see a real spray. Split into two correctly-shaped detections:

> **Fixed:** a spray is **one source, many accounts, few tries each** — grouped by
> `source_ip` over distinct usernames (Rule 5a). E012's 25 failures against one
> account from one IP is **failed brute force**, not spray, and now lives in Rule
> 5b (`T1110.001`). The reseeded dataset adds a real spray campaign (one IP walking
> 10 accounts) so 5a has something true to catch.

### Rule 6 — Inactive Account Activity · `T1078.002`

Any activity on a non-Active account — deliberately no time or success filter.

> **Note:** the missing detection-window filter here is **intentional** (any
> activity on a terminated account matters, whenever it happened) and is now
> documented as such in the view. Owns the E016 case end to end.

E016 (terminated) received two failed US attempts, then a **success from Russia**
(95.213.45.67). Technique is **T1078.002 (Domain Accounts)** — corrected from the
exec summary's earlier T1078.004 (Cloud Accounts); this is on-prem domain auth.

## Phase 3: Triage and Prioritization

The consolidated queue is the live `v_triage_queue` view — every row carries its
ATT&CK technique and any static threat-intel match on the source IP. This table
is generated, so the "reproduce with `SELECT * FROM v_triage_queue`" instruction
is actually true (it was not, before). See
[results/DETECTION_OUTPUT.md](results/DETECTION_OUTPUT.md).

### Threat-intel enrichment

Source IPs are matched against a static `ip_enrichment` table that **mirrors
ADTE's offline intel ranges** (`adte/intel/_mock.py`) — so this lab and the triage
engine name the same infrastructure identically, with zero API keys:

| Source IP | Names as | Seen in |
|---|---|---|
| 185.220.101.45 | `tor-exit` (mock-tor-list, conf 0.85) | E004 brute force |
| 45.33.32.201 / .156 | `scanner` (mock-scanner-feed, conf 0.75) | spray / failed brute force |
| 95.213.45.67 (RU) | *unenriched* — documented feed gap | E016 success |

This replaces the previous logic, which called an IP "suspicious" purely because
it wasn't in `10.x`.

## Severity model and the path to ADTE

This lab uses flat HIGH/MEDIUM/LOW severities. That is deliberately the **manual
precursor** to ADTE, whose limitations — no additive signal stacking, no numeric
bands, no verdict policy — motivated building a deterministic 0–100 engine. The
mapping onto ADTE's `classify_verdict` bands:

| Lab severity | ADTE verdict band | Illustrative ADTE scoring |
|---|---|---|
| HIGH | `high_risk` (score > 70; Critical ≥ 90) | E016 RU success: impossible-travel-class + IP reputation + hour anomaly + cluster context → ≥ 75 |
| MEDIUM | `medium_risk` (30–70) | Unfamiliar country alone → 30–50 |
| LOW | `low_risk` (< 30) | After-hours by IT staff → ~10 |

Where the lab emits one label per rule, ADTE stacks weighted signals into a
single score — which is exactly the gap this lab makes visible.

### The handoff, made concrete

The mapping table above is not hypothetical: `scripts/adte_bridge.py` runs
every user this lab flags through ADTE's actual engine, **fully offline**
(static intel table, no API keys), and writes the comparison to
[docs/ADTE_HANDOFF.md](docs/ADTE_HANDOFF.md). Highlights from the real run:
the lab's deliberate benign FP (E017) scores 8/low_risk — the engine agrees
with ground truth; E016's terminated-account takeover scores 48 with its
`ip_reputation` at **zero** (the RU source is an offline-feed gap, visible in
the numbers); and E012's *failed* brute force — MEDIUM in the lab — is ADTE's
**top score (62)**, because signal density (scanner IP + failure burst + odd
hour) outweighs single-rule impact. Manual flags in, deterministic scores and
routing out: that is the workflow this lab exists to motivate.

## Phase 4: Data Integrity Validation

Containment decisions made on a corrupt inventory isolate the wrong device.
Validate before acting.

### Orphan devices

`v_orphan_devices` → **DEV-099**, listed owner E999, who does not exist in the
directory. Cannot be attributed, notified, or verified.

### Inventory mismatch — the wrong-device trap

`v_inventory_mismatch` → **DEV-011**: primary inventory says Karen Taylor (E011),
backup says Grace Lee (E007). **E007 is the subject of the impossible-travel
finding.** If containment is driven off the wrong inventory, you either isolate
an uninvolved employee's device or miss the one tied to a live investigation.
The two inventories must be reconciled by hand before any action — a
`test_integrity.py` case pins the E007-is-also-the-travel-subject link so this
scenario can't silently disappear in a refactor.

### Active users without devices

`v_users_without_devices` → E017–E020: active employees with no assigned device
(previously computed but never surfaced). E016 is correctly **excluded** — it's
terminated, not an active user missing a device.

**Tuning notes** (the qualitative layer the numbers don't capture):

| Rule | Tuning note |
|---|---|
| Unfamiliar Country | Cross-reference HR travel; the no-baseline branch is the FP source, by design |
| Impossible Travel | Velocity gate + `travel_exceptions`; centroid distance overstates border hops |

## Development & AI-assisted methodology

Detection logic, ground-truth labels, and metrics are deterministic and
human-owned. False-positive validation followed Trail of Bits' `fp-check`
three-phase methodology (data flow → attacker-meaningfulness → impact), applied
by hand — the plugin itself was not installed in this environment, and rather
than assert a run that didn't happen, FP rate was **measured** directly against
ground truth. That review confirmed one candidate detection (E017's no-baseline
flag) as an operational-robustness artifact rather than a true positive — retained
and documented as the measured precision cost, not silently dropped — and surfaced
three defects in the pre-overhaul logic (a brute-force counting bug, an
impossible-travel false positive, a spray coverage gap), all fixed here. Full
write-up: [docs/SECURITY_REVIEW.md](docs/SECURITY_REVIEW.md).
